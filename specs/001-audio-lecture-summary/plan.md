# Implementation Plan: Audio Lecture Summary Application

**Branch**: `001-audio-lecture-summary` | **Date**: 2026-03-21 | **Spec**: [specs/001-audio-lecture-summary/spec.md]
**Input**: Feature specification from `specs/001-audio-lecture-summary/spec.md`

## Summary

Build a modular web application for capturing audio (recording or upload), transcribing it using the KKY ASR API, and generating summaries via the Ollama KKY LLM service. The architecture follows a stateless backend (FastAPI) and a modern React frontend (Desktop-first) with a swappable adapter pattern for AI services to ensure modularity and future extensibility.

## Technical Context

**Language/Version**: Python 3.12+, TypeScript 5.4+  
**Primary Dependencies**: FastAPI, Pydantic, React 18, Vite, Lucide-React, `httpx`, `fpdf2`, `markdown2`  
**Storage**: Stateless (Server); Session-based recovery via Browser IndexedDB (Frontend)  
**Testing**: pytest (Backend), Vitest + React Testing Library (Frontend)  
**Target Platform**: Linux/Docker (Backend), Modern Browsers (Frontend - Desktop-first Responsive)  
**Project Type**: Web Service + Single Page Application  
**Performance Goals**: AI task progress updates every 5s; PDF generation < 2s; UI interaction < 100ms  
**Constraints**: No server-side user database; Czech language hardcoded for MVP; Supporting document processing deferred.  
**Scale/Scope**: MVP prototype for individual lecture transformation

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

- **I. Modularity**: Swappable adapters for `ASRProvider` and `LLMProvider` MUST be implemented.
- **II. Simplicity**: Backend MUST remain stateless; no user auth database required for MVP.
- **III. Quality-First**: Unit tests for transcription/summarization logic and adapter contracts are mandatory.
- **IV. Robustness**: UI MUST implement stage-based status text (Uploading, Transcribing, etc.).
- **V. Scalability**: Architecture MUST support horizontal scaling of the stateless API.
